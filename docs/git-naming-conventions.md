# Git Naming Conventions

This document defines the recommended formats for common Git names in this repository, covering the most common collaboration naming scenarios for branches, commits, tags, PR titles, remotes, and stashes.

## General Rules

- Prefer English, ASCII, lowercase letters, numbers, and hyphens.
- Use kebab-case for multiple words, for example `openagent-oauth-flow`.
- Names should describe intent. Do not use only `new`, `update`, `wip`, `temp`, or `final`.
- Avoid spaces, underscores, mixed case, special characters, and personal names.
- When an existing package name or directory name can express the scope, prefer the existing name, for example `cli`, `codex-plugin`, `openclaw-plugin`, or `docs`.

## Branch Names

Use this format for regular branches:

```text
<type>/<short-kebab-summary>
```

Common `type` values:

- `feat`: new feature
- `fix`: bug fix
- `docs`: documentation change
- `chore`: maintenance change
- `refactor`: behavior-preserving refactor
- `test`: test-related change
- `ci`: GitHub Actions or CI configuration
- `build`: build, packaging, or dependency configuration
- `release`: release preparation branch
- `hotfix`: urgent fix branch
- `spike`: exploratory experiment branch; before merging, rename it to a formal type or close it

Recommended examples:

```text
feat/openagent-oauth-standardization
fix/codex-plugin-version-sync
docs/git-naming-conventions
ci/release-workflow-token-check
```

Not recommended:

```text
feature
fix_bug
WIP-login
ray-test
update
```

## Commit Messages

Commit messages should follow the Conventional Commits style:

```text
<type>(<scope>): <summary>
```

Omit `scope` when there is no clear scope:

```text
<type>: <summary>
```

Common `type` values should stay aligned with branch types:

- `feat`
- `fix`
- `docs`
- `chore`
- `refactor`
- `test`
- `ci`
- `build`
- `perf`
- `revert`

Recommended `scope` values:

- `cli`
- `codex-plugin`
- `openclaw-plugin`
- `docs`
- `release`
- `deps`

Recommended examples:

```text
feat(openclaw-plugin): add brokered oauth session exchange
fix(codex-plugin): sync manifest version during release
docs: add git naming conventions
ci(release): verify npm scope access before publish
chore: release packages
```

Commit summary rules:

- Use present-tense imperative mood, for example `add`, `fix`, or `remove`.
- Start with a lowercase letter and do not end with a period.
- Keep the summary short. Put complex context in the commit body.
- Use `!` for breaking changes and include `BREAKING CHANGE:` in the body.

Breaking change example:

```text
feat(cli)!: require explicit server url

BREAKING CHANGE: CLI calls now require --server-url when no default is configured.
```

## Pull Request Titles

PR titles should reuse the commit message format when possible:

```text
<type>(<scope>): <summary>
```

Recommended examples:

```text
docs: add git naming conventions
fix(openclaw-plugin): refresh auth after protected mcp failure
ci(release): harden package publish workflow
```

When multiple commits are combined into one PR, the PR title should describe the final result that is visible to users or important to maintainers, instead of listing all intermediate steps.

## Tags

Release tags should be generated by the Changesets and GitHub Actions release process, or follow the requirements of that process. Do not manually move, overwrite, or rename tags that have already been published.

This repository is a monorepo and defaults to independent package releases. Package-level release tags use the full npm package name and version:

```text
<package-name>@<major>.<minor>.<patch>
```

Use this format for prerelease versions:

```text
<package-name>@<major>.<minor>.<patch>-<channel>.<number>
```

Recommended examples:

```text
@call-e/cli@0.1.0
@call-e/codex-plugin@0.1.0
@call-e/openagent@0.1.0
@call-e/codex-plugin@0.2.0-rc.1
```

When referencing package-level tags that contain `@` and `/` in installation commands or scripts, prefer an explicit quoted `--ref`, for example:

```bash
codex plugin marketplace add CALLE-AI/call-e-integrations \
  --ref '@call-e/codex-plugin@0.1.0' \
  --sparse .agents/plugins \
  --sparse packages/codex-plugin/plugin
```

Use repository-level version tags only when the entire repository uses one unified version and the tag represents a repository-level release:

```text
v<major>.<minor>.<patch>
```

Use this format for prerelease versions:

```text
v<major>.<minor>.<patch>-<channel>.<number>
```

Recommended examples:

```text
v1.2.3
v1.3.0-rc.1
v1.3.0-beta.2
```

Avoid tagging a single package release as a generic `v<version>`, because tag names are unique within one repository. If multiple packages each release `0.1.0`, generic version tags create ambiguity or conflicts.

## Remote Names

- `origin`: the default remote for the current repository.
- `upstream`: the upstream repository in fork-based workflows.

Avoid personal names or temporary meanings as remote names. When a temporary remote is needed, use a descriptive name and delete it after the work is complete.

Recommended examples:

```text
origin
upstream
integration-test
```

## Stash Names

Stashes should include a clear description, using this format:

```text
<type>(<scope>): <summary>
```

Recommended examples:

```bash
git stash push -m "docs: draft git naming conventions"
git stash push -m "fix(openclaw-plugin): inspect auth retry state"
```

Avoid undocumented stashes, because it becomes hard to decide later whether they can still be restored or deleted.

## File And Directory Names In Git

- Documentation files should use kebab-case, for example `git-naming-conventions.md`.
- Do not casually rename existing directory names for packages, plugins, skills, and similar project areas.
- Before changing marketplace entry points, plugin names, visible labels, or installation commands, read `docs/agent-integration-layout.md`.
